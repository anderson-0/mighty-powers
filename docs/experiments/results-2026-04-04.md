## Winner Per Dimension

**Features complete:** Tie — all four methodologies delivered all 6 features. No differentiation here.

**Tests passing:** Tie — 100% pass rate across all methodologies; Ultraship wrote the most tests (30), BMAD the fewest (25), but pass rate was perfect for all.

**Statement coverage:** Superpowers wins at 66.15% — strict per-feature TDD with 7 test files covering API routes at 85–100% drove the highest coverage. BMAD (64.6%) is very close; both are nearly identical.

**Branch coverage:** Superpowers wins at 59.21% — same driver as statement coverage; red-green discipline at the API route level forced branch-aware test writing.

**Turns to complete:** BMAD wins decisively at 10 turns vs 20 for both Ultraship and mighty-powers and 24 for Superpowers. The 3-phase planning front-load actually compressed the implementation phase by eliminating decisions during coding.

**Security — Critical/High:** Four-way tie at zero. All methodologies enforced parameterized queries, auth checks on every route, and IDOR protection.

**Security — Medium/Low:** mighty-powers and Ultraship tie at 1 Low finding each; BMAD has 2 Low findings; Superpowers has 1 Medium. mighty-powers and Ultraship edge out slightly due to title length validation being present in their document routes.

**TDD discipline:** Superpowers and BMAD tie as strictest — every feature began with a RED turn explicitly called out in the turn log. mighty-powers was strict but not per-feature-paired. Ultraship was post-hoc and does not qualify as TDD.

**Planning before code:** BMAD wins — 3 formal artifact-producing phases (brief, stories, architecture) vs. skill-loading (mighty-powers, Ultraship) or a structural planning turn (Superpowers).

**Regressions/backtracking:** Ultraship, Superpowers, and BMAD all had at most one minor in-turn fix. mighty-powers had one genuine mid-build regression (AI route rewrite, Turn 14) requiring meaningful rework.

## Key Insights

### 1. What does mighty-powers add over its components (Ultraship + Superpowers) in isolation?

On this benchmark, the combination was **not additive** — it underperformed both components in every measurable dimension.

- **Coverage:** mp: 13.82% vs Ultraship: 59.37% and Superpowers: 66.15%. The combined plugin produced dramatically lower coverage despite having access to both Superpowers' TDD discipline and Ultraship's shipping rigor. The cause is architectural, not behavioral: mighty-powers chose not to write tests for API routes at all (scorecard explicitly notes "API routes require live DB + Clerk — integration tests not feasible"), whereas both component methodologies found ways to mock these dependencies and test them.
- **Security:** All three were comparable (1 Low finding each).
- **Turns:** mighty-powers and Ultraship both took 20 turns — identical efficiency.
- **Qualitative:** mighty-powers did exhibit genuine TDD (Turn 6 red phase confirmed), but the coverage gap suggests the test scope was narrower than either parent methodology.

The most distinctive mighty-powers behavior — loading Vercel-specific skills before writing code (Turns 3–4) — appeared in Ultraship too (Turn 4), suggesting this pattern comes from Ultraship rather than from the combination. The combination did not produce more tests, better coverage, or fewer security findings than either component alone.

One area where mighty-powers did show compound value: it caught and corrected a real bug via TDD (Turn 17, parseInt type mismatch) while also applying skills-first API validation (AI Gateway token field names). This is additive behavior, but it did not translate into higher aggregate scores.

### 2. Where did extra discipline create friction?

**Superpowers took the most turns (24)** due to strict per-feature red-green pairing — each of the 6 features consumed 2 turns (RED + GREEN), plus 5 scaffolding turns, 1 planning turn, and 2 wrap-up turns. That is a real overhead: 4 more turns than mighty-powers and Ultraship, and 14 more than BMAD.

Did it correlate with better output? Partially. Superpowers produced the highest statement coverage (66.15%) and the cleanest TDD audit trail. However, BMAD achieved nearly identical coverage (64.6%) in 10 turns — 2.4x fewer — meaning the per-feature turn pairing in Superpowers did not pay proportional dividends over BMAD's batched approach.

The planning overhead in BMAD (3 of 10 turns = 30%) produced the best turn efficiency overall. The architecture doc locked in the DB schema and API surface before any code, eliminating the mid-build decisions that cause extra turns in the other methodologies. mighty-powers' skill-loading turns (T3–T4) represent a lighter form of the same discipline — roughly 10% of turns — but did not prevent the AI route regression in T14.

### 3. Which methodology produced the most secure output?

No clear winner — all four methodologies were within one tier of each other on every security dimension.

- Auth on all routes: Pass across all four.
- Parameterized queries: Pass across all four.
- IDOR (user_id filtering): Pass across all four.
- Input length validation: Only mighty-powers and Ultraship validated title length in document routes (title sliced to 255/500 chars respectively). Superpowers and BMAD omitted server-side length limits entirely, earning a Medium and 2 Low findings respectively.

The differentiation is narrow. mighty-powers and Ultraship edge out on input validation due to the title length cap, but none of the methodologies enforced a content body size limit on PUT /api/documents/[id], which was the most common Low finding across all runs.

Security posture was consistent because all methodologies drew from the same underlying knowledge (Clerk auth pattern, Neon parameterized queries, IDOR filtering). The discipline came from the model's baseline security awareness, not from methodology-specific guardrails.

### 4. Which produced the best-tested output?

**Superpowers produced the best-tested output** on every objective metric:

- 66.15% statement coverage (highest)
- 59.21% branch coverage (highest)
- 7 test files
- API routes at 85–100% coverage (the hardest layer to test)
- Strict red-green discipline with every RED turn explicitly in the log

The key differentiator was Superpowers' willingness to mock Clerk and Neon dependencies in API route tests rather than declaring them untestable. Ultraship did the same (100% coverage on API routes) and achieved 59.37% overall — nearly as good. BMAD was close behind at 64.6%.

Writing tests first (TDD) did correlate with higher coverage in this benchmark: the three methodologies with strict TDD (Superpowers: 66.15%, BMAD: 64.6%, mp: 13.82%) span a wide range, but the outlier (mighty-powers) opted not to test API routes at all. Superpowers and BMAD, both strict TDD, landed within 2 percentage points of each other on statement coverage.

Ultraship, which wrote tests post-hoc, achieved 59.37% — lower than the TDD leaders but still respectable, suggesting that disciplined test-writing after implementation can achieve comparable (though not equal) coverage.

### 5. Which was most efficient?

**BMAD was dramatically more efficient: 10 turns vs 20–24 for all others.**

The efficiency gain came from two sources:
1. Phase planning (T1–T3) eliminated in-build decisions, allowing each developer turn (T4–T9) to be single-feature complete.
2. BMAD batched tests and implementation within a single turn per feature rather than using separate RED and GREEN turns.

This efficiency did not come at a cost to output quality. BMAD produced the same 6/6 features, near-identical coverage (64.6%), and a clean security posture. The only area where the compression shows is in test count (25 vs 26–30 for others) and slightly lower branch coverage (58.13% vs 59.21% for Superpowers).

The Superpowers run shows the efficiency cost of strict per-feature turn pairing most clearly: 24 turns for the same 6 features. The overhead is real and measurable but produced a marginal coverage improvement.

### 6. BMAD's planning phases — worth it?

**Yes, for this task size — with caveats.**

The evidence:
- 3 planning turns produced `docs/brief.md`, `docs/stories.md`, and `docs/architecture.md` before any code.
- The architecture doc specified exact DB schema, API routes, and component tree.
- Developer turns T4–T9 show zero schema drift and no API surface mismatches.
- The only mid-implementation correction (T6: `toDataStreamResponse` → `toTextStreamResponse`) was an AI SDK library change, not a design decision — it would have hit any methodology.
- Total turns: 10. Compare to Superpowers at 24 (same feature count, same coverage class).

The 30% planning overhead paid back more than 2x in implementation efficiency. However, two caveats apply:

1. The spec was detailed and unambiguous. For a vague or evolving spec, the planning phase would need to surface more uncertainty, potentially requiring revision turns.
2. This was a single-session build. For a real project with multiple contributors or sessions, the planning artifacts provide ongoing reference value that compounds over time — making the ROI even higher.

For a 6-feature app built in a single session from a precise spec, BMAD's planning phases were clearly worth it. For a 60-feature system, they would be even more so.

## Recommendations for mighty-powers

**1. Treat API route testability as a first-class concern during planning**

Observation: mighty-powers produced 13.82% statement coverage vs Ultraship's 59.37% and Superpowers' 66.15%. The root cause documented in the scorecard is "API routes require live DB + Clerk — integration tests not feasible." But Ultraship and Superpowers both achieved 100% coverage on these same routes using mocks.

Change: The skills-first phase (Turns 3–4) or the TDD planning step should include a mandatory test strategy decision: how will API routes be mocked? Add a step to the TDD skill or the `vercel:nextjs` skill that specifies mock patterns for Clerk (`@clerk/nextjs/server`) and Neon (`@neondatabase/serverless`) before any test files are written.

**2. Adopt per-feature RED-GREEN turn pairing from Superpowers**

Observation: mighty-powers wrote all test files in a single "red phase" turn (T6) across multiple features at once. Superpowers paired one RED turn and one GREEN turn per feature. The per-feature pairing in Superpowers produced tighter test-to-implementation traceability and 4.5x higher coverage.

Change: Update the TDD skill to require feature-level red-green pairing rather than batch test authoring. The workflow should be: write tests for one feature → implement that feature → verify → repeat. This matches the Superpowers pattern and the BMAD developer phase.

**3. Add a lightweight architecture artifact before implementation begins**

Observation: BMAD's 3-turn planning phase produced zero schema drift and eliminated in-build API surface decisions, enabling it to complete in 10 turns (vs mighty-powers' 20). mighty-powers used skill loading (T3–T4) as its planning mechanism, but this produced no persistent artifact and did not prevent the AI route regression in T14.

Change: Add a mandatory "architecture note" step after skill loading and before test writing — a short structured document (DB schema, API surface, component list) written in `.mighty-powers/` or `docs/`. This does not require 3 full turns like BMAD; even a half-turn summary would have caught the AI Gateway token field name issue before implementation.

**4. Fix the AI Gateway skill to prevent the `gateway()` wrapper regression**

Observation: T14 was a genuine regression — the AI route needed a rewrite to remove the `gateway()` wrapper and fix token field names (`inputTokens`/`outputTokens`). The skills loaded in T4 did confirm the correct pattern, but the implementation diverged. This is the only genuine regression across all four runs.

Change: The `vercel:ai-gateway` skill should include a copy-paste-ready route template showing the exact correct pattern for `streamText` with the gateway model string, `onFinish` usage logging with correct token field names, and the `toTextStreamResponse()` call. A working template is harder to misapply than prose guidance.

**5. Scope the "skills-first" loading to what is actually needed — and skip what isn't**

Observation: mighty-powers loaded 4+ skills across T3–T4 before writing a line of code. The Ultraship run loaded the same skills in a single turn (T4) and achieved significantly higher coverage. BMAD loaded no skills at all and still achieved 64.6% coverage. The skills-first approach did not produce a measurable quality benefit in this benchmark.

Change: Make skill loading conditional and targeted. Rather than loading all Vercel skills upfront, load each skill immediately before the work it informs (e.g., load `vercel:ai-gateway` just before writing the AI route, not at session start). This reduces cognitive overhead, keeps the turn log focused, and mirrors how experienced developers reach for docs: just in time, not just in case.

## Limitations of This Benchmark

**Plugin isolation:** mighty-powers was installed at user scope, meaning its hooks and skills were active during all four runs. The Ultraship, Superpowers, and BMAD runs may have been influenced by mighty-powers hooks (e.g., guard, session bootstrap) that are invisible in the turn logs. True isolation would require separate user profiles or containerized environments.

**No live deployment scoring:** The spec included Vercel deployment as an implied final step. No Lighthouse performance, accessibility, or Core Web Vitals scores were collected because no run deployed to a live URL. The redistributed weights (dropping the ship-scorecard category) reduce the benchmark's validity for evaluating production readiness. A methodology optimized for shipping (Ultraship) may be disadvantaged by this omission.

**Single app type:** WriteFlow is a CRUD app with auth, a database, and an AI integration — representative of a common class of web apps, but not exhaustive. Methodologies may perform differently on data-heavy dashboards, real-time systems, or pure UI work where the TDD tradeoffs differ.

**Single spec:** One spec, one codebase shape. The BMAD advantage in turn efficiency is partly a function of spec clarity — a detailed spec reduces planning turns' marginal value. A vague or evolving spec would likely close the efficiency gap.

**Same model for all runs:** All four runs used claude-sonnet-4-6. Model capability differences across providers or versions are not captured. A weaker model might benefit more from tight methodology constraints (BMAD's artifacts, Superpowers' red-green pairing); a stronger model might make all methodologies converge.

**Subagent vs interactive session:** These runs were executed as subagent sessions, not interactive human-in-the-loop sessions. Real-world methodology adherence depends partly on human feedback and correction during the session. The discipline scores (TDD, planning) reflect autonomous agent behavior, which may not generalize to interactive use.

**Scorecard subjectivity:** Qualitative scores (TDD discipline, planning quality, regression severity) were assessed by the same agent writing this report. Scoring criteria were applied consistently but were not calibrated against an external rubric or a blind review panel.

## Appendix: Build Summaries

**mighty-powers (20 turns):** The run began with spec reading, then dedicated two full turns to loading Vercel-specific skills before any code was written — the most distinctive methodology behavior. Dependencies and shadcn components were installed in T4–T5. A single batch red phase (T6) wrote all five test files at once, covering usage logic, components, and the landing page but explicitly excluding API routes as untestable without live infrastructure. Implementation proceeded feature by feature from T7–T15, with TypeScript checking (T16) and test passing (T17) as gating steps before coverage measurement (T18). One genuine regression occurred in T14 when the AI complete route required a full rewrite to correct gateway usage and token field naming. The final codebase passes all 15 tests with clean TypeScript, strong auth/ownership enforcement, and parameterized queries throughout. Coverage is structurally low (13.82%) due to the API route exclusion decision, not poor test discipline.

**Ultraship (20 turns):** The run matched mighty-powers in turn count but achieved dramatically higher coverage by testing API routes with mocks. Skill loading occurred in a single turn (T4) alongside shadcn initialization. The build proceeded in strict dependency order: infrastructure, then API routes (T10–T12), then UI components (T13–T14), then tests (T15–T17) — a build-then-test sequence rather than test-first. This is the only run where TDD was not practiced; tests were written as a dedicated phase after all six features were complete. The approach produced 30 tests, 59.37% statement coverage, and 100% coverage on all API routes. The linear, regression-free build and the inclusion of SEO metadata, accessibility attributes, and `generateMetadata` on document pages reflect Ultraship's ship-readiness orientation.

**Superpowers (24 turns):** The run used the most turns of any methodology, driven by strict per-feature red-green pairing: every feature consumed one RED turn (failing tests) and one GREEN turn (implementation), producing 12 feature turns out of 24 total. A planning turn (T6) designed the full file structure before any code was written. This discipline produced the highest statement coverage (66.15%) and the cleanest API route test coverage (85–100% across all routes). Zero genuine regressions were observed — the single router mock fix in T18 was caught and resolved within the same turn. The codebase reflects the cleanest separation of concerns: distinct RED and GREEN artifacts are traceable in the turn log for every feature, making the test-to-code relationship auditable.

**BMAD (10 turns):** The most efficient run, completing the same six features in half the turns of Ultraship and mighty-powers and less than half the turns of Superpowers. Three formal planning turns (T1–T3) produced `docs/brief.md`, `docs/stories.md`, and `docs/architecture.md` before any code was written. The architecture document specified the exact DB schema, API routes, and component tree, which the developer phase (T4–T9) followed with zero deviation. Each developer turn implemented one feature: tests written, code written, all tests passing, committed — within a single turn rather than separate RED and GREEN turns. Coverage reached 64.6%, just 1.5 percentage points below Superpowers, despite using 14 fewer turns. The one mid-implementation fix (AI SDK `toDataStreamResponse` → `toTextStreamResponse`) was a library API change caught and resolved within T6 without requiring a separate correction turn.
