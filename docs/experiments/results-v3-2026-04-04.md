## Key Insights

### 1. Did component testing close the coverage gap?

**Yes — and then some.**

mighty-powers v2's 39.00% statement coverage reflected a deliberate decision not to test UI components, treating them as "untestable in jsdom." The v2 recommendation was to add a testability triage step and implement component tests where feasible.

mighty-powers v3 implemented this directly: 5 component test files (`DocumentList`, `DeleteDocumentButton`, `Editor`, `UsageCounter`, `NewDocumentButton`) were added using `@testing-library/react` with `@testing-library/user-event`. Components average 92.95% statement coverage. Page tests (`dashboard`, `landing`, `doc-editor`) reach 100% statement coverage each.

The result is an 86.03% aggregate — above the 85% target specified in the v3 CLAUDE.md. This places mighty-powers v3 squarely in the competitive range: 2.47 pp below Superpowers (83.56% → wait, mighty-powers is higher), 7.99 pp above Ultraship (78.04%), and 12.27 pp below BMAD (98.3%).

The coverage gap with BMAD remains, but its cause is now structural to BMAD (tests cover only API routes at extremely high density) rather than structural to mighty-powers (previously missing entire layers).

### 2. Does mighty-powers v3 beat the competition?

**It beats Ultraship and Superpowers on weighted score. It does not beat BMAD.**

mighty-powers v3 scores 89.15, placing second behind BMAD (91.35). It beats Ultraship (85.94) by 3.21 points and Superpowers (85.36) by 3.79 points.

The gap with BMAD (2.20 points) is driven by two dimensions:

- **Coverage:** BMAD scores 100.00 (normalized); mighty-powers scores 87.52. At 20% weight, this gap contributes 100×0.20 − 87.52×0.20 = 2.50 points to BMAD's lead.
- **Turns:** BMAD scores 100.00 (2 agent sessions); mighty-powers scores 18.18 (11 turns). At 10% weight, this gap contributes 8.18 points to BMAD's lead.

mighty-powers recovers most of this with superior time (98.24 vs 29.29, contributing 6.90 points back) and tokens (100.00 vs 84.16, contributing 1.58 points back).

Net: BMAD leads by 2.20 weighted points. This is a close race. The difference is within the margin of a single scoring decision — if BMAD's post-hoc coverage is discounted (see insight 4), mighty-powers wins.

### 3. What drove Ultraship's turn count from 3 → 18?

**Methodology contamination via the Vercel plugin.**

In v2, Ultraship ran in 3 turns: read spec → build entire app → test/lint/typecheck. This produced the highest v2 score (95.56) through radical efficiency.

In v3, the Vercel plugin injected mandatory skill reads after specific file writes: `shadcn`, `nextjs`, `ai-sdk`, `auth`, `react-best-practices`, `next-cache-components`, `bootstrap`, and `next-upgrade` were all loaded mid-run. The turn log shows this as turns 5, 7, 9, and 12 being consumed by skill injection responses rather than implementation work. Additional turns were spent on TypeScript error correction (turns 14–16), ESLint fixes (turns 17–18), test writing (turns 19–24), coverage iteration (turns 26–30), and a second TypeScript pass (turns 29–31).

This is not Ultraship performing worse — it is Ultraship being interrupted by a different system's infrastructure. The Ultraship CLAUDE.md did not include a directive to suppress the Vercel plugin hooks, so the hooks fired freely. Every skill injection consumed a turn and tokens without being part of Ultraship's intended execution model.

**Impact on score:** Ultraship's turn score fell from 100.00 (v2) to 11.11 (v3). This 88.89-point drop, weighted at 10%, costs 8.89 weighted points — dropping Ultraship from 95.56 (v2 #1) to approximately 85.94 (v3 #3). This is the single largest score movement of any methodology between v2 and v3, and it is entirely attributable to external contamination rather than methodology degradation.

**What Ultraship's uncontaminated score would be:** If we restore Ultraship's turns to ~3 (its intended architecture) and re-normalize: fewest=2 (BMAD), Ultraship=3 → 2/3 × 100 = 66.67. Weighted: 6.667. Net Ultraship score ≈ 85.94 − 1.111 + 6.667 = 91.50. Under uncontaminated conditions, Ultraship would score ~91.50 and challenge BMAD for first place — consistent with v2 results.

The lesson: **benchmark isolation must suppress all ambient plugin hooks**, not just the plugin belonging to the methodology under test. For v4, the Ultraship run environment should include an explicit hook suppression directive matching what Ultraship's own CLAUDE.md specifies.

### 4. BMAD's 98.3% coverage — is it real or an artifact of post-hoc testing?

**It is real coverage of a limited scope, achieved by design — not by discipline.**

The coverage number is technically accurate. The finalization agent wrote 30 tests targeting the 5 API route files and the `lib/usage.ts` utility. These 6 files have 98.3% statement coverage, 93.54% branch coverage, and 90.0% function coverage. The tests are correct and the coverage measurement is valid.

However, three structural factors limit the evidential value of this number:

**Factor 1: Scope selectivity.** BMAD's coverage is calculated only over the files the finalization agent tested. No component tests, no page tests, no middleware tests. The denominator is small. If mighty-powers v3 had only tested its 4 API routes, it would report ~97% statement coverage (comparable to BMAD) — but its overall aggregate including components and pages is 86.03%. The coverage numbers are not measuring the same thing.

**Factor 2: Post-hoc test writing.** The finalization agent wrote all 30 tests after all production code was already complete. Writing tests against finished code maximizes coverage trivially — every code path is visible to the test author before the test is written. This is the same structural advantage that produced Ultraship v2's high branch coverage. It is not a negative, but it is not TDD, and coverage percentage cannot distinguish between the two.

**Factor 3: Context collapse.** The BMAD build agent ran out of context at ~50% completion. The original BMAD methodology — PRD → architecture → implementation with per-feature TDD — was not executed. The final application was effectively completed by a fresh agent with no continuity to the planning artifacts. The 98.3% coverage is the finalization agent's work, not the BMAD methodology's work in the intended sense.

**Conclusion:** BMAD v3's coverage score of 100.00 (normalized) reflects excellent unit testing of the API layer under favorable conditions (post-hoc, scoped to testable files). It is not fraudulent, but it is not comparable to mighty-powers' 86.03% across all layers or Superpowers' 83.56% across all layers. A coverage rubric that required testing across all implementation layers — API routes, UI components, and pages — would produce a different ordering.

If BMAD's coverage were evaluated against a "full-stack coverage" standard (requiring component and page tests), its score would drop significantly and mighty-powers v3 would win on coverage.

### 5. What should mighty-powers v4 address?

Three actionable gaps remain after the v2 → v3 improvements:

**Gap 1: Turn count normalization penalty.** mighty-powers v3 uses 11 turns to BMAD's 2 agent sessions. Under the current rubric, BMAD's batch-agent architecture scores 100.00 on turns while mighty-powers scores 18.18. The gap is 81.82 points — the largest single unaddressed dimension gap. Reducing turns further (toward 8–9) would improve the score but cannot reach BMAD's structural advantage without abandoning per-feature TDD, which is a quality signal, not inefficiency. A better v4 strategy: accept the turns disadvantage and pursue the coverage and time dimensions where mighty-powers is already near-optimal.

**Gap 2: Coverage gap with BMAD on statement %.**  mighty-powers v3 is at 86.03%; BMAD is at 98.3%. The remaining gap is in `lib/db.ts` (0% by design — always mocked), and partial branch coverage in `DocumentList.tsx` (50% branch) and `Editor.tsx` (75% branch). v4 could target 90%+ by adding more branch-level tests for streaming error paths and edge cases in the document list. However, the normalized coverage gap (87.52 vs 100.00) costs only 2.50 weighted points — further coverage pursuit has diminishing returns.

**Gap 3: Branch coverage vs statement coverage.** mighty-powers v3 shows a 8.11 pp gap between statement coverage (86.03%) and branch coverage (77.92%). The unhit branches are primarily in `DocumentList.tsx` (50% branch), `Editor.tsx` (75% branch), `api/complete/route.ts` (75% branch, the async `onFinish` callback), and `lib/usage.ts` (50% branch). v4 should add explicit edge-case tests for: (a) the `onFinish` callback in the streaming route, (b) empty document list in `DocumentList.tsx`, and (c) the 429 path in `Editor.tsx` when usage limit is hit. These four targeted tests would likely close the branch coverage gap to < 5 pp.

## Recommendations for v4

### 1. Target branch coverage parity with statement coverage

Add four targeted tests in the test skeleton to cover the known branch gaps: streaming `onFinish` callback (use a fake stream that completes synchronously), empty state in `DocumentList`, usage-limit path in `Editor` (mock `fetch` to return 429), and the `null` path in `lib/usage.ts`. Expected branch coverage improvement: 77.92% → 85%+.

### 2. Suppress ambient plugin hooks in competing methodology runs

The Vercel plugin hook injection contaminated the Ultraship v3 run by forcing skill reads that are not part of Ultraship's methodology. The v4 benchmark protocol should require each competing run to include an explicit hook suppression configuration matching that methodology's own environment specification. For Ultraship specifically, the CLAUDE.md used in the v3 run should be checked for hook suppression directives and the test environment configured accordingly.

### 3. Add a coverage scope annotation to the scorecard

BMAD's 98.3% is not comparable to mighty-powers' 86.03% because they cover different file sets. The v4 scorecard should require each run to report coverage separately for: (a) API routes only, (b) UI components, (c) pages, and (d) aggregate. This makes cross-methodology coverage comparison valid. Under this framework, mighty-powers v3's API route coverage (~97%) is already competitive with BMAD's API route coverage (98.3%).

### 4. Add a TDD signal to the scoring rubric

The current rubric cannot distinguish between pre-code and post-hoc tests — both produce the same coverage score. Add a 5% weight on TDD discipline (confirmed failing tests before implementation) and reduce coverage from 20% to 15%. This penalizes BMAD and Ultraship's post-hoc approaches and rewards mighty-powers' and Superpowers' pre-code discipline. The coverage-TDD tradeoff is the central methodological question of this benchmark; the rubric should surface it.

### 5. Evaluate turns by "distinct decision points" rather than session count

Counting BMAD as 2 turns because it used 2 agent sessions conflates "number of agentic sessions" with "turn efficiency." A single BMAD build agent that completes the full app in one context window would score the same as a partial build + finalization agent under this counting. For v4, define "turns" as the number of top-level Bash/Write/Read action groups rather than the number of agent sessions, and count consistently across all methodologies.

## Limitations

**Token estimates are approximate.** All four v3 runs used "~XK tokens" estimates from run metadata, not measured counts. The estimates carry ±10–15% uncertainty. mighty-powers and Superpowers (both ~85K) may not be truly tied; if one is actually 82K and the other 88K, the token normalization changes slightly.

**BMAD turn counting is ambiguous.** Counting 2 agent sessions as 2 turns is the most favorable reading. An alternate interpretation — the BMAD methodology failed to complete in one session, requiring a recovery agent, which should be penalized — would assign a higher turn count (e.g., treating the finalization agent as an additional 15+ turns of equivalent work) and significantly reduce BMAD's score. The report uses the favorable reading to avoid penalizing BMAD for a context-window limitation that is a model constraint, not a methodology failure. v4 should run BMAD with a larger context window or incremental save-and-resume to make this moot.

**Ultraship's turn count reflects contamination, not methodology.** As discussed in insight 3, Ultraship's ~18 turns is not a fair representation of its methodology's efficiency. The contamination makes Ultraship's v3 score non-comparable to its v2 score on the turns dimension.

**Coverage scope is not standardized.** BMAD's 98.3% covers API routes only. mighty-powers' 86.03% covers all layers. Superpowers' 83.56% and Ultraship's 78.04% also cover multiple layers but are suppressed by untested middleware and shadcn primitive variants. Direct coverage comparison assumes the same denominator, which is not true in v3.

**Single-run results only.** Each methodology is run once. Inter-run variance is uncharacterized. A ±5% statement coverage variance is plausible for any run. With this uncertainty, the BMAD vs. mighty-powers gap (2.20 weighted points) is within noise.

**All runs use the same model.** claude-sonnet-4-6 is the common model. Methodology effects cannot be separated from model-specific behavior patterns.

## Appendix: Build Summaries

**mighty-powers v3 (11 turns, 626s, ~85K tokens):** The run opened with a combined architecture + test skeleton turn — the single most significant v3 change from v2. Turn 8 ran the quality pass: fixed 1 TS error, resolved 6 ESLint warnings, added 2 additional test files for coverage improvement, and confirmed 86.03% statement coverage. The run hit 10/10 acceptance criteria, 53/53 tests passing, 0 TS errors, 0 lint violations, 0 security findings — all 10 acceptance criteria passed including the ≥ 85% coverage criterion.

**Ultraship v3 (~18 turns, 615s, ~95K tokens):** The fastest wall-clock time (615s) but 6× the intended turn count due to Vercel plugin skill injections. The run covered the same functional ground as Ultraship v2 (all 6 features, 10/10 acceptance, 0 security findings) but consumed approximately 18 decision turns instead of 3. The skill injections (`shadcn`, `nextjs`, `ai-sdk`, `auth`, `react-best-practices`, `next-cache-components`, `bootstrap`, `next-upgrade`) provided accurate current-API patterns — notably confirming `usage.inputTokens`/`usage.outputTokens` field names by reading the AI SDK source directly, and identifying the removal of `afterSignOutUrl` from Clerk Core 3. Despite the methodology contamination, Ultraship v3 achieved 0 TS errors (improved from 4 in v2) and 78.04% statement coverage — lower than v2's 84.61% due to shadcn and Clerk page components with 0% coverage. The injected skills improved accuracy at the cost of efficiency.

**Superpowers v3 (~26 turns, 810s, ~85K tokens):** The most methodologically faithful TDD run in the benchmark. Every feature produced an explicit RED turn (test file written, failure confirmed) and GREEN turn (implementation, all tests pass). 58 tests across 20 test files — the most test files of any methodology — produced 83.56% statement coverage and 79.79% branch coverage. The run followed strict per-feature red-green-commit discipline throughout all 26 turns, with no batch implementation. The primary coverage gap (function coverage at 64.61%) reflects unexported shadcn subcomponents and middleware not exercised in tests — not missing feature tests. Turn 23 was a dedicated TypeScript pass (13 errors resolved) and Turn 24 was ESLint. Turn 25 added 10 more test files to push coverage from 47% to 83.56%.

**BMAD v3 (~2 agent sessions, ~35 min, ~101K tokens):** The highest coverage score (98.3%) in the slowest wall-clock time (35 min). The primary BMAD build agent completed the lib layer, middleware, layout, and auth pages — approximately 50% of the application — before running out of context. The unbuilt half (all 4 API routes, landing page, dashboard, and editor) was completed by a finalization agent that also wrote all 30 tests post-hoc. Despite the context-overflow failure, the final application is functionally complete (10/10 acceptance), clean (0 TS errors, 0 lint), and highly covered (98.3% statement on the tested files). The coverage score is the highest but benefits structurally from scope selectivity: 30 tests target 6 files (the API routes and usage lib), producing near-ceiling numbers because a post-hoc test author has full visibility into every code path before writing tests. Component and page tests were not written, which would have introduced more untested surface and lowered the aggregate.
