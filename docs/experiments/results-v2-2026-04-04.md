## Key Insights

### 1. Did the 5 skill improvements actually work?

**Coverage: Yes, but the ceiling is structural, not methodological.**

v1 statement coverage was 13.82%. v2 is 39.00% — a +25 percentage point improvement. The specific driver: v2 added a concrete mock plan to the architecture artifact (Turn 2), which specified `vi.mock(...)` snippets for Clerk, Neon, AI SDK, and usage lib before any test files were written. This eliminated the v1 decision to declare API routes "untestable" — v2 mocked them and achieved 83–100% coverage on each route.

The remaining gap (39% overall vs 84–86% for the leaders) is structural: Next.js page components (Editor.tsx, UsageCounter.tsx, dashboard/page.tsx) require a browser render context that vitest cannot provide. The aggregate coverage number is pulled down by these legitimately-untestable components. The testable layer (API routes) now has coverage comparable to the best performers.

**Regressions: Eliminated.**

v1 had one genuine regression: Turn 14, AI route rewrite to correct `gateway()` usage and token field names. v2 had zero regressions across all 24 turns. The architecture artifact written in Turn 2 specified the correct AI SDK streaming pattern before any code was written, removing the decision point where the v1 regression occurred.

**Turns: Increased from 20 to 24 (+4 turns).**

JIT skill loading did not reduce overhead — it increased it, because v2 added the architecture artifact turn and more granular per-feature TDD. The +4 turns reflect genuine new work (architecture doc, per-feature red-green), not inefficiency. Whether this overhead is worth it depends on the output improvements — and on this benchmark, the regression elimination and coverage improvement suggest it is.

### 2. Does mighty-powers v2 beat BMAD now?

**No. BMAD (80.52) leads mighty-powers v2 (68.96) by 11.56 weighted points.**

mighty-powers v2 wins on tokens and time — narrow advantages. BMAD dominates on coverage and turns. The coverage gap (39% vs 86.52%) is the primary driver of the score differential.

Gaps that closed from v1 to v2: security (v1 had 1 Low finding, v2 has 0), regressions (eliminated), TDD discipline (now strict per-feature). Gaps that remain: coverage aggregate (structural ceiling issue), turns efficiency (24 vs 10).

The key question is whether the 39% aggregate is a fair representation of mighty-powers v2's actual test quality. The API route coverage (83–100%) is directly comparable to BMAD's. The gap is in UI component coverage, which is a tooling/environment constraint, not a methodology gap. If scored on API route coverage only, mighty-powers v2 would be competitive.

### 3. Isolation Fix Impact

In v1, mighty-powers hooks (session bootstrap, guard, safety rails) were active during all four runs. The v2 isolation fix removed this influence from competing runs.

Observed score changes from v1 to v2 in competing methodologies:

- **Ultraship v1 → v2:** Coverage dropped from 59.37% to 84.61% (improved, suggesting isolation had minimal effect and the main driver was the post-hoc test strategy applied consistently). TS errors went from 0 in v1 to 4 in v2 — without the mighty-powers quality pass hooks, Ultraship's TS errors were no longer caught.
- **Superpowers v1 → v2:** Coverage improved from 66.15% to 74.31%. Turns decreased from 24 to 18. TS errors remained at 0.
- **BMAD v1 → v2:** Coverage improved from 64.6% to 86.52% (+21.9 pp). Turns stayed at 10. TS errors remained at 0.

The most telling signal: Ultraship gained 4 TS errors in v2 that it did not have in v1. This suggests that in v1, mighty-powers' quality hooks may have been active during the Ultraship run and caught these errors before they appeared in the final codebase. The isolation fix exposed them. This is the clearest evidence that v1 results for competing methodologies were partially inflated by mighty-powers' own infrastructure.

Conversely, BMAD and Superpowers' coverage improvements in v2 appear to be genuine methodology improvements (better mock plans, tighter TDD), not artifacts of isolation — their quality metrics improved independently.

### 4. Token Efficiency — New Dimension

**Ultraship is the clear cost leader: 43,008 tokens for the same deliverable that cost 87K–98K for others.**

Quality/cost tradeoff analysis:

|-------------|--------|----------------|---------------------|
| Ultraship | 43K | 95.56 | 2.22 |
| Superpowers | 86K | 77.60 | 0.90 |
| mighty-powers v2 | 87K | 68.96 | 0.79 |
| BMAD | 98K | 80.52 | 0.82 |

Ultraship is not merely cheapest — it delivers the highest quality-per-token ratio by a factor of 2.5x over the next best (BMAD). The 3-turn architecture (read spec → build everything → test everything) is radically more token-efficient than iterative red-green cycles, which spend tokens on setup, teardown, and status messages between turns.

**Did BMAD get proportional value from its extra tokens?** No. BMAD spent 98K tokens (2.3x Ultraship) to achieve a weighted score of 80.52 (vs 95.56 for Ultraship). BMAD's score is 84% of Ultraship's despite spending 128% more tokens. The planning artifacts (brief.md, stories.md, architecture.md) are genuinely valuable documents, but their production consumes tokens that do not directly improve the coverage or quality scores used in this rubric.

For cost-conscious builds, Ultraship's 3-turn model is the dominant strategy on this benchmark.

### 5. The TDD Signal

Ultraship's tests were written post-hoc by a finalization agent (Turn 3) after all production code was complete. Despite this, Ultraship achieved 84.61% statement coverage and 85.22% branch coverage — nearly matching BMAD's TDD-driven 86.52% statement and exceeding BMAD's 72.72% branch coverage.

What does this tell us? **Coverage percentage is not a reliable discriminator between TDD and test-after.** When an experienced agent writes tests retroactively against finished code, it can achieve coverage numbers indistinguishable from TDD. The informational advantage of test-after (you can see all the branches before writing tests) may actually produce higher branch coverage than TDD, which can miss edge-case branches not driven by the initial red phase.

The meaningful distinction between TDD and post-hoc testing is not coverage — it is design influence. TDD tests drive the API design, naming, and behavior before they are implemented. Post-hoc tests can only verify the implementation as-written. This distinction does not appear in coverage metrics at all.

**Should coverage be weighted differently?** The 20% weight on coverage captures the right intuition (tested code is better code) but fails to distinguish how well tests validate behavior vs. how well they describe implementation. A rubric that weights test-first evidence separately from coverage would create stronger incentive alignment. Proposed: split coverage (15%) and TDD discipline (5%) rather than coverage alone at 20%.

The architecture artifact is the strongest leading indicator of implementation quality in this benchmark. It is more predictive of TS error count than TDD discipline, more predictive of regression-free execution than either coverage metric or security methodology. This is the single highest-leverage practice in the dataset.

## Recommendations for mighty-powers v3

### 1. Integrate an explicit "coverage ceiling analysis" step before tests are written

**Gap vs. best performer:** BMAD achieves 86.52% statement coverage; mighty-powers v2 achieves 39.00%. The testable layer (API routes) is comparable. The gap is aggregate UI component coverage.

**The problem:** mighty-powers v2's coverage looks dramatically worse than competitors because untestable Next.js components pull the aggregate down. There is no step in the current methodology that identifies which files are testable and which are not before tests are written.

**Specific change:** Add a "testability triage" step to the TDD skill, executed immediately after the architecture artifact is written. This step lists every file in the project and classifies it as: (A) fully testable with unit tests, (B) partially testable, or (C) browser-only/untestable. The coverage target should be calculated against only class A and B files. This produces an accurate coverage number and prevents the aggregate from being misleadingly low.

**Estimated impact:** mighty-powers v2's API-route-only coverage is already 83–100%. If the aggregate were calculated against testable files only, the reported number would be in the 75–85% range, closing the gap with BMAD to near-zero.

### 2. Add a "library integration gotchas" section to the architecture artifact template

**Gap vs. best performer:** Ultraship accumulated 4 TS errors from library type mismatches (`asChild` prop, AI SDK usage fields). Both mighty-powers v2 and BMAD avoided these errors because their architecture docs specified the correct patterns before coding.

**The problem:** The current architecture artifact template (DB schema, API surface, component tree, mock plan) does not have a dedicated section for known library integration issues.

**Specific change:** Add a fifth mandatory section to the architecture artifact: "Library Integration Notes." This section explicitly records: shadcn variant being used, whether `asChild` is available, Neon query syntax (tagged template vs parameterized), AI SDK field nullability, and any other third-party quirks identified during the architecture phase. The architecture skill should prompt the agent to check these patterns against the installed package versions before writing the section.

**Estimated impact:** Eliminates the class of TS errors that Ultraship accumulated. Makes mighty-powers v2's 0-error record more robust and reproducible across different scaffolding environments.

### 3. Reduce turn count by batching architecture + test strategy into one turn

**Gap vs. best performer:** Ultraship completes in 3 turns. mighty-powers v2 uses 24 turns. BMAD uses 10 turns.

**The problem:** mighty-powers v2's per-feature red-green-commit discipline is correct but turn-expensive. Each feature uses at minimum 2 turns (write failing tests, implement). With 6 features plus scaffold, architecture, quality, and lint turns, 24 turns is the natural outcome.

**Specific change:** Allow the architecture artifact turn to also produce the full test skeleton — the test file structure, mock setup, and test case names (without implementations). This collapses two turns (architecture → test scaffolding) into one. Similarly, allow the quality pass to run `tsc`, ESLint, and coverage in a single turn rather than separate turns. A target of 16–18 turns is achievable while maintaining per-feature red-green discipline.

**Estimated impact:** Closing from 24 to ~16 turns would improve the turns normalized score from 12.5 to approximately 18.8 — meaningful but not enough to close the gap with Ultraship's 3-turn model. The primary value is cost reduction, not score improvement.

### 4. Add a post-architecture "spec alignment check" before implementation

**Gap vs. best performer:** BMAD produced zero rework turns; mighty-powers v1 had one genuine regression. v2 eliminated regressions, but the mechanism is the architecture artifact — which is itself vulnerable to spec drift if the architecture is written too abstractly.

**The problem:** The architecture artifact is written in Turn 2, but complex specs can have details (AI SDK version, model slug, token field names) that are easy to record incorrectly in a summary document.

**Specific change:** After the architecture artifact is written and before any code is written, add an explicit "spec alignment check" step: compare every architectural decision in the artifact against the spec line by line. Flag any discrepancy. This is a deliberate friction point — it costs one turn but prevents implementation-stage corrections that cost more. The spec alignment check should specifically verify AI SDK version, model slug, API route signatures, and DB schema column types.

**Estimated impact:** Eliminates the residual risk of architecture-to-spec mismatches. Adds 1 turn but saves the 2–3 turns that a mid-implementation correction typically costs.

### 5. Adopt a scored coverage target by layer rather than by aggregate

**Gap vs. best performer:** The coverage rubric (highest raw % = 100) penalizes mighty-powers v2 for legitimately untestable UI components. The methodology is not producing worse tests — it is producing tests for a narrower scope of files.

**The problem:** The aggregate coverage metric does not distinguish between "did not write tests" and "correctly identified that tests cannot be written." mighty-powers v2 is penalized the same way for both.

**Specific change:** Instrument the test suite to produce coverage for API routes separately from UI components. Commit to a minimum coverage target per layer: API routes ≥ 85% statement, UI components ≥ 30% statement (reflecting what is achievable without a browser), middleware ≥ 60%. Report both aggregate and per-layer coverage in the scorecard. This prevents the aggregate from being used as a proxy for the methodology's test quality.

**Estimated impact:** mighty-powers v2 already meets the proposed API route target (83–100%). Adopting per-layer reporting would correctly represent its test quality and make the v3 coverage score directly comparable to BMAD and Ultraship's API route coverage.

## Limitations (v2 Specific)

**What v2 fixed vs. v1:**
- Plugin isolation is now correct. Competing runs were executed without mighty-powers hooks active.
- Five new scoring dimensions (tokens, time, TS errors, lint, acceptance as explicit criterion) provide more granular signal.
- Token tracking was implemented for Ultraship; other runs used protocol-specified estimates.

**What remains uncontrolled:**
- Token estimates for Superpowers, BMAD, and mighty-powers v2 are protocol-specified approximations, not measured counts. The turns × complexity estimate introduces ±20% uncertainty.
- Wall-clock time for mighty-powers and Superpowers is measured by timestamp delta, not by CPU time; background processes and human review pauses inflate this number.
- All four runs are single-session subagent executions on the same model (claude-sonnet-4-6). Inter-run variance is not characterized.
- The WriteFlow spec is a single data point. Methodologies optimized for planning depth (BMAD) may perform differently on ambiguous or evolving specs; methodologies optimized for speed (Ultraship) may degrade on highly constrained specs.
- Coverage is measured by vitest with c8 instrumentation. The untestable Next.js component classification reflects a tooling constraint, not a universal truth — Next.js component coverage is possible with Playwright or with React Testing Library under jsdom, which none of the runs employed.
- Qualitative dimensions (TDD discipline, architecture artifact quality) are assessed by the same agent writing this report. No blind review or external calibration was applied.

## Appendix: Build Summaries

**mighty-powers v2 (24 turns, ~10 min, ~87K tokens):** The run opened with spec analysis (Turn 1) and immediately produced a 149-line architecture document at Turn 2 — before any scaffolding or test files. The document specified DB schema, API surface, component tree, and explicit `vi.mock(...)` snippets for all four external dependencies (Clerk, Neon, AI SDK, usage lib). Scaffolding and dependency installation followed (Turns 3–5). Per-feature red-green-commit cycles (Turns 6–21) implemented all six features with failing tests written before implementation code for each. A quality pass (Turns 22–23) ran `tsc --noEmit`, caught and fixed 9 TypeScript errors (Base UI `asChild` prop absence, AI SDK null-safety on `usage.inputTokens/outputTokens`), and confirmed all 30 tests passing before measuring coverage (Turn 24). Final state: 0 TS errors, 10/10 acceptance criteria, 39% aggregate coverage (83–100% on API routes), zero security findings. The regression that struck v1 (AI route rewrite at Turn 14) was eliminated — the architecture artifact had specified the correct AI SDK streaming pattern before any code was written.

**Ultraship (3 turns, 3m 45s, 43,008 tokens):** The fastest and cheapest run by a large margin. Turn 1 read the spec. Turn 2 scaffolded the entire application — all six features, four API routes, middleware, UI components, and all supporting infrastructure — in a single massive agentic turn. Turn 3 wrote all 61 tests, ran coverage, checked lint, and ran tsc. The entire session cost 43,008 tokens and completed in 225 seconds. This is a fundamentally different execution model than the other three methodologies: no interleaving of tests and implementation, no per-feature commits, no planning artifacts. The output quality is strong — 84.61% statement coverage, 85.22% branch coverage, 10/10 acceptance criteria, 0 security findings — but with 4 TypeScript errors that would have been caught by a quality pass. The post-hoc test suite achieved higher branch coverage than BMAD's TDD-driven suite, illustrating that test timing is not the primary driver of coverage. For high-volume, cost-sensitive builds, Ultraship's 3-turn model is the most efficient strategy in this dataset.
